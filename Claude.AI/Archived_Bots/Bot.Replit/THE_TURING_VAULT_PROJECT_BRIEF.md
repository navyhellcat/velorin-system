1. WHAT IT IS
The Turing Vault is a full-stack web platform where users take blind tests to determine whether a passage of text was written by a human or generated by AI. It is built around the premise that as AI writing becomes indistinguishable from human writing, the ability to detect it is a rare, measurable, and increasingly valuable skill.
The platform collects and stores human response data — every guess, every confidence level, every piece of typed reasoning — creating a proprietary dataset that grows with every user session. This data is the platform's core long-term asset.

3. CORE FEATURES
Game Engine
* Three modes: Standard (easy/medium passages), Hard (harder passages mixed), Adversarial (intentionally deceptive AI passages)
* Weighted scoring: Standard 1.0×, Hard 1.35×, Adversarial 1.75×
* Passage selection: Server-side algorithm picks balanced human/AI passages per mode ratio, avoids recently-seen passages per visitor
* Word limit: All passages truncated to 70 words at display time (full text preserved in DB). Sentences cut mid-way show ...

Thresholds shift by mode: Hard −3%, Adversarial −6% per tier.

Bot Detection (invisible)
* Passage serve time recorded server-side when each question loads
* On answer submit, elapsed time calculated:
   * < 8 seconds → flagged rejected
   * 8–16 seconds → flagged suspicious
   * 16 seconds → flagged clean
* Client receives identical response regardless of flag — bots cannot detect they are being caught
* All flags visible in admin Research terminal

5. THINGS TO NEVER DO AGAIN
These are hard-learned lessons from building this system. Future agents must read this.
1. Never run destructive SQL without a SELECT test first
A bulk UPDATE using a regex wiped 523 passage bodies by collapsing them to single lines. The regex [^\n]* matched the entire flat body string. Always test on one row with SELECT inside a transaction before running any UPDATE at scale.
2. Never assume passage bodies contain newlines
Passage bodies are stored flat — no line breaks. Any regex using \n anchors as separators will match the entire string. Treat each body as a single long line.
3. Wrap all bulk DB operations in a transaction
Every multi-row UPDATE or DELETE must be wrapped in BEGIN; ... ROLLBACK; first to verify output, then COMMIT only when confirmed correct.
4. The valid Anthropic model name is claude-haiku-4-5-20251001
Not claude-3-haiku-20240307 or any other variant. The ingestion pipeline will silently fail or error with an invalid model name. This caused multiple failed ingest jobs early in the build.
5. The DB push command is cd lib/db && pnpm run push
Not drizzle-kit push from root. Not pnpm db:push. Always run from the lib/db directory. After any schema change, run this before restarting the API server.
6. formatPercent() expects 0–100, not 0–1
The frontend utility assumes values are already in percentage scale. Backend returns accuracy as 0–100. Do not multiply by 100 before passing to this function.
7. Resend free tier only sends to verified emails
On the free Resend plan, outbound emails only go to addresses verified in the Resend dashboard. Before public launch, verify a custom domain (alerts@turingvault.com or similar) so the storage alert system works reliably.
8. Never change primary key column types
Switching between serial and varchar/UUID on existing tables generates destructive ALTER TABLE statements that break all foreign key relationships and existing data.
9. The API server port is 8080
The frontend proxies /api to localhost:8080. If you change the API server port, the Vite proxy config must be updated to match.
10. leaderboard returns data.alltime[] not data[]
The leaderboard hook returns an object with an alltime key. Always destructure correctly: topBoard?.alltime?.slice(0, 10).

6. ARCHITECTURE DECISIONS WORTH PRESERVING
* Passage sequence stored in session status field (active:id1,id2,id3): avoids a separate session_passages junction table. Works but is fragile — a future refactor should move this to a proper table.
* Visitor tracking via cookie (not auth): anonymous by design. Users don't need accounts. The visitors table tracks return visitors for passage de-duplication.
* Bot timing is server-side only: client never knows about timing. This is intentional and must stay that way. Never add any client-side timing logic or countdown UI.
* Passage truncation at render, not storage: the full body is preserved in the DB. The 70-word limit is applied only at display time in test.tsx via truncatePassage(). This means future changes to word limits require no DB changes.
* Mode-based verdict scaling uses a t(base) offset helper: Hard mode subtracts 3% per tier threshold, Adversarial subtracts 6%. This is in results.tsx getVerdict().

7. WHAT IS NOT YET BUILT
These are intentional gaps — known future work:
* Authentication / user accounts: Currently all anonymous. Adding accounts would enable persistent score history and unlock subscription billing.
* Rate limiting per visitor: Sessions can be created without limit. Should add per-IP or per-visitor session caps.
* Cloudflare Turnstile on leaderboard submission: Discussed but not implemented. Leaderboard names can currently be bot-submitted.
* Passage quality scoring: No mechanism to flag or rate passage quality. Admins cannot mark a passage as "too obvious" or "too ambiguous" without deactivating it entirely.
* Analytics dashboard depth: The public analytics page exists but is basic. Deeper breakdowns (accuracy by topic, by word count, by source) are possible with the existing data.
* Session passage table refactor: The active:id1,id2,... pattern in the status column should be replaced with a proper session_passages table.
* Offsite data archiving: At 370K sessions the auto-export fires, but there is no automated push to S3 or external DB. This requires external credentials (AWS, Supabase, etc.).

8. PASSIVE INCOME ROADMAP
Ranked by estimated effort vs. return:
A. Adversarial Mode Paywall (fastest to ship, ~3–5 hours)
Gate adversarial mode behind a $5–8/month subscription. Standard and Hard remain free forever — the difficulty cliff makes adversarial feel premium. Use Stripe (Replit has a Stripe integration). This is a recurring revenue model with near-zero marginal cost per user.
What to build: Stripe checkout session on session creation when mode === "adversarial", subscription check middleware, subscriber badge on leaderboard.
B. Research Dataset Sales (zero extra build, high ceiling)
The responses table is already the product. Once you have 10,000+ completed sessions, the dataset (passage + user judgment + confidence + rationale + bot flag + mode) is sellable to:
* AI safety researchers (few-shot human baseline data)
* NLP companies building detection tools (GPTZero, Originality.ai competitors)
* Academic institutions (IRB-safe anonymous behavioral data)
* Kaggle Datasets marketplace (exposure + licensing fees)
What to build: A data licensing page, a sample dataset download, and outreach to 3–5 research labs.
C. "Detect AI" API (medium build, subscription SaaS)
Use the corpus and crowd response patterns to offer a text classification API. Developers submit text, receive back a human_probability score. Charge $0.001–0.01 per call or $15–30/month for a plan. Competing services (GPTZero, Originality.ai, Writer.com) charge exactly this range.
What to build: A new Express route /api/public/classify, API key management, usage metering, Stripe billing.
D. B2B Media Literacy Licensing (highest per-deal value, longest sales cycle)
Universities, newsrooms, fact-checking organizations, and corporate L&D teams will pay $500–5,000/year for a white-labeled or team-branded version. The pitch: "Train your journalists / employees to detect AI-generated content."
What to build: Team/organization accounts, custom leaderboards, usage reporting, bulk session caps per org.
E. Affiliate & Sponsorship (passive, low effort)
Weekly or monthly "AI Detection Report" newsletter summarizing anonymized aggregate stats ("This week, only 38% of users correctly identified adversarial climate-change content"). Grow a subscriber list and monetize with AI-adjacent sponsors (writing tools, AI companies, cybersecurity).
What to build: Email list signup on results page, newsletter template, Beehiiv or Substack integration.

9. FOR THE CEO BOT
The platform's core differentiation is not the game mechanic — it's the proprietary behavioral dataset. Every human who plays is labeling data. The longer the platform runs, the more valuable the corpus becomes. Do not rush to monetize in ways that reduce user volume (heavy paywalls, aggressive friction). Optimize for data collection first, monetization second.
The ideal growth loop: Free tier drives volume → volume grows dataset → dataset quality enables API product → API revenue funds marketing → marketing drives more volume.
The platform is genuinely unique. No other public tool gamifies human-AI text detection at this level while collecting research-grade behavioral data simultaneously.
