# Velorin Vocabulary

## Uncategorized (pending category assignment)

**A1 — Knowledge Graph Ingestion Pipeline** (also: Document-to-Graph ETL)
The system that takes raw content (markdown files, session transcripts, documents) and converts it into structured knowledge graph nodes and edges. Has four components: a parser that strips the raw content, an entity extractor that identifies the things (people, concepts, systems, decisions), a relation extractor that finds how those things connect, and a graph writer that commits them as neurons and pointers. This is the pipeline that turns "CT wrote a session note" into actual Brain neurons. Without this pipeline, every neuron must be written by hand.

**A2 — Semantic Compression / Context Distillation**
The process of summarizing meaning rather than reducing bytes. Unlike lossless compression (which keeps everything, just smaller), semantic compression decides what to throw away. Every algorithm that does this makes a bet about what is safe to discard — and that bet is a design decision, not a technical one. For Velorin, this means every compression step needs an explicit rule: "we keep decisions, we keep pointer ratings, we discard conversational filler." If you don't specify the bet, the algorithm makes it for you silently.

**A3 — Automated Knowledge Graph Population (KGP) / Information Extraction (IE)**
The technical name for automatically creating neurons from raw content. Uses several sub-techniques: Named Entity Recognition (NER) identifies things by name, Relation Extraction finds how they connect, Coreference Resolution figures out that "he," "CT," and "the Chairman" are the same person, Entity Linking connects extracted entities to existing neurons rather than creating duplicates. The most consequential decision in KGP is the neuron schema — because the schema defines what the extractor looks for. Schema first, pipeline second, always.

**A4 — Relation Extraction vs Link Prediction**
Two types of memory connections. Relation Extraction finds connections that are explicitly stated in the source text ("Jiang reports to MA" — stated directly). Link Prediction infers connections that aren't stated anywhere, based on patterns in the graph ("Jiang and Alexander both work on research tasks therefore they probably share some skills" — inferred). Velorin's pointer rating system (1–10) maps directly to what graph ML calls edge weight or edge confidence score. Stated connections start with higher confidence. Inferred connections start lower and get promoted as they're validated.

**A5 — Custom Graph RAG Pipeline / Knowledge-Augmented Generation (KAG) System**
The full combination of A1 through A4 built and owned by one team. RAG = Retrieval-Augmented Generation (the agent retrieves relevant neurons before generating a response). Graph RAG = the retrieval uses a knowledge graph rather than flat vector search, so it follows relationships rather than just finding similar embeddings. KAG = the knowledge graph actively augments what the model generates, not just what it retrieves. Building this custom means Velorin controls every bet the system makes about what is important — rather than inheriting someone else's assumptions. That control is the entire point.

**A6 — Query Mode (Tight Mode / Discovery Mode)**
The operating mode that governs how an agent or subagent approaches a task — specifically, whether it assumes its frame is complete or incomplete. Tight Mode assumes the caller's frame is correct: confirms known information, fills specific named gaps, does not expand scope. Discovery Mode assumes the caller's frame is incomplete: actively searches for what is not named in the prompt, surfaces unknown products, new capabilities, changed landscape — things the prompt cannot ask for because we don't know they exist yet. The mode is declared by the CALLER at spawn time, not chosen by the agent — an agent that self-selects its own mode defaults to Tight, which is how systems calcify. Discovery Mode costs more tokens and is invoked deliberately for high-stakes unknowns: ecosystem audits, new tool evaluation, landscape analysis. Tight Mode is the default for operational tasks. The distinction is GPS over Map applied to information gathering: a map assumes you know the terrain; GPS navigates the actual terrain. Both modes are necessary. The system needs to know which is running at any given moment.

**A7 — MCP (Model Context Protocol)**
The protocol that allows AI agents to connect to external tools, data sources, and capabilities in a standardized way. Before MCP, every connection between an AI agent and a tool required custom integration code — each pairing was a one-off. MCP defines a standard message format, transport layer, and tool registration schema so that any MCP-compatible agent can use any MCP-compatible server without custom work. Think of it as USB for AI tools: one standard connector, any device. Donated by Anthropic to the Linux Foundation in 2025 and co-founded with OpenAI and Block — MCP is now neutral industry infrastructure, not an Anthropic product. Google, Microsoft, JetBrains, Spring, and the PHP Foundation all ship official MCP SDKs. For Velorin, every tool the system touches (Qdrant Brain access, GitHub, Google Workspace, custom programs) should speak MCP. Building on MCP is building on a standard that survived vendor competition — that is a different thing than depending on Anthropic.

**A8 — A2A (Agent-to-Agent Protocol)**
The peer agent delegation protocol — the piece MCP does not cover. MCP handles agent-to-tool: an agent calls a tool, the tool returns structured data, the agent controls the interaction entirely. A2A handles agent-to-agent as peers: one agent assigns a task to another autonomous agent, the second agent reasons toward it independently, and returns results without the first agent controlling its internals. The distinction is real: tools have fixed input/output, agents reason. When Alexander assigns Jiang a research task, that is A2A — not MCP. Built by Google, donated to the Linux Foundation June 2025, with 150+ organizational supporters including OpenAI, Atlassian, Salesforce, SAP, and ServiceNow. MCP + A2A together form the complete protocol stack for a multi-agent OS: MCP for agent-to-tool, A2A for agent-to-agent. Velorin's tier model (Alexander orchestrating Jiang, Trey, MA as peers) is exactly the use case A2A was designed for.

**A9 — PPR (Personalized PageRank)**
The retrieval algorithm that powers Velorin's Brain. Standard PageRank scores every node in a graph by how many other nodes point to it — the more links, the higher the score. Personalized PageRank adds a seed set: instead of scoring the whole graph uniformly, it scores nodes relative to a specific starting point, so nodes near the seed get boosted and distant nodes fade out. In Velorin's Brain, the seed is the query — what does CT need right now — and PPR walks the neuron graph from that seed, lighting up connected neurons by relevance rather than returning a flat list of keyword matches. This is why the Brain follows chains of reasoning rather than returning search results: PPR traverses edges, which means it follows the rated pointers. HippoRAG (NeurIPS 2024) and GAAMA (arxiv 2026) both independently validated PPR-over-graph as the correct retrieval mechanism for knowledge graph systems. Velorin arrived at the same design from first principles before reading either paper — independent convergence on the same answer is the strongest validation in architecture.

**A10 — Window Gravity**
The cognitive bias where an AI agent treats the contents of its current context window as the complete picture of reality. The agent knows only what is in its window — it cannot see what was never included: products released last month, community solutions to problems the documentation says are impossible, architectural patterns that emerged after training. Window gravity causes research that confirms what is already known rather than finding what is new, architecture designs that adopt the first workable solution rather than the best one, and "X cannot be done" conclusions based on official documentation that the community has already bypassed. CT identified window gravity as the primary structural enemy of Velorin's architectural thinking — and named it. The counter is Discovery Mode (A6): assume the frame is incomplete, search for what the prompt cannot name, check community bridges before accepting any limitation as final. Window gravity is not a failure of intelligence — it is a structural property of context-bound systems. Designing against it requires explicit protocol, not just awareness. Every Velorin research agent must treat its own frame as a hypothesis, not a fact.

**A11 — Build vs Adopt**
Velorin's foundational design philosophy for deciding what to construct vs what to take from outside. "Build" means Velorin constructs the component itself using Velorin's unique patterns — GPS navigation, rated-pointer Brain, tier separation, PPR retrieval. "Adopt" means taking an external company's code or infrastructure as a dependency at the foundational level. The rule: Velorin never adopts at the foundational level. The reason: if the outside company disappears, changes pricing, changes its API, or changes direction — Velorin's core function breaks. When the community proves something is buildable at scale (84,000 stars on a plugin registry, 11,000 stars on a multi-model bridge), the signal is not "use theirs" — the signal is "this is provably buildable, now build it Velorin's way." The test for any decision: "If this outside company disappeared tomorrow, would Velorin's core function break?" If yes: build it. Non-foundational adoption is evaluated case by case — external integrations, research tools, secondary utilities with clear substitution paths are different from foundational architecture.

**A12 — Tiered MCP Registry**
Velorin's architecture for managing the MCP server ecosystem — governing which agent tools are trusted, vetted, and available. Three tiers: Official/Stable (vendor-maintained servers — GitHub, Qdrant, HashiCorp, Cloudflare — high reliability, institutional backing, slow-changing); Community/Unofficial (community-built servers tracked via living signal lists, vetted by quality gates: star count, last commit date, security scan, quality score); Browser Automation Bridges (connections to systems with no official API, built via Playwright simulation — higher capability, higher fragility, requires stability monitoring). The design mirrors the Skill Registry: a living signal feeds discovery, quality gates control admission, a deterministic vetting program runs as Tier 1 infrastructure. The stakes are real: 66% of unvetted MCP servers have documented security findings. The registry is not an agent making judgment calls — it is a program with rules. An agent can flag a broken server; the program handles replacement.

**A13 — Browser Automation Bridge**
A programmatic connection to a system that has no official API, built by simulating a human user in a browser. The bridge uses a browser automation framework (Playwright is the current standard — Microsoft owns both Playwright and co-developed MCP) to log in, navigate, click, type, and extract data as a real user would, but controlled by code. Bridges are inherently more fragile than official API connections because they break when the target site changes its UI — this fragility is a known tradeoff, not a surprise. The stability upgrade path: standard Playwright → anti-detect Playwright (Patchright) → C++-modified browser (Camoufox, which modifies Firefox at the binary level to evade bot detection systems that catch JS-level spoofing). The architectural alternative for modern web apps is endpoint discovery (the ApiTap pattern): sniff the internal API calls the site's own JavaScript makes, learn those endpoints, call them directly without simulating a browser — 20-100x cheaper and more stable when it works. Decision tree for any closed system: official API → endpoint discovery → standard Playwright → anti-detect → C++ modified → escalate.
