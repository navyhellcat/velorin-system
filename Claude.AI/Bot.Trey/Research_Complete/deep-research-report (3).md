# Executive Summary  
*FlashMoE* is a system that offloads inactive MoE experts to NVMe storage and brings them into unified memory on demand【133†L65-L72】【124†L96-L104】.  It uses a learned cache policy (mixing recency and frequency) to decide which experts stay resident【133†L65-L72】. On Apple Silicon (e.g. MacBook/Studio), FlashMoE “trusts the OS”: only the few active experts per token are loaded via asynchronous I/O, and macOS’s page cache implicitly manages eviction【124†L96-L104】. In practice, a 4-bit 397B-parameter MoE (209GB on disk, 17B active) runs at ~4.4 tokens/s on a 48GB MacBook Pro (M3)【131†L1-L4】. We estimate a 64GB M2 Ultra (800GB/s memory) could handle ~300–400B models (4-bit) at similar or slightly higher rates (~5 tokens/s), and an M4 Ultra (192GB) up to ~500B, with throughput up to ~7–10 tokens/s when warm. Beyond this, SSD bandwidth (∼16–20 GB/s on Apple) and CPU/GPU compute limit scaling (Confidence ≥ 75%). 

## NVMe Offload Mechanism  
FlashMoE’s key innovation is **expert-layer swapping**: the full MoE is stored on SSD, and only the needed experts are loaded into RAM. According to the paper, “inactive experts” are offloaded to SSD and reloaded on demand【133†L65-L72】. Specifically, FlashMoE performs parallel `pread()` calls for the small active expert slices (e.g. 4 experts × ~6.75 MB each) using Grand Central Dispatch【124†L96-L104】. The rest of the model remains on NVMe (e.g. 209 GB for 4-bit quantized 397B model), and macOS’s page cache automatically brings pages into unified memory when accessed【124†L96-L104】. This is a “zero-overhead” strategy: no custom eviction logic is needed on Apple – the OS does LRU-style caching at the page level. In effect, FlashMoE’s ML-based policy (from the paper) is implemented via the operating system’s cache replacement. The result is a ~71% SSD hit rate in practice【124†L96-L104】. In contrast to naive schemes (e.g. LRU), FlashMoE reports a *cache hit rate improvement up to 51%* over LRU/LFU on generic hardware【133†L65-L72】. (High confidence: 90% – this matches both the arXiv abstract and the Apple blog.)  

## Apple Silicon Unified Memory Performance  
Apple Mx series uses unified memory (UMA) accessible by CPU/GPU. For example, an M2 Ultra Mac Studio has **64–192 GB unified memory** and *800 GB/s* bandwidth【129†L79-L84】. FlashMoE leverages this by streaming experts through the shared memory controller【124†L100-L104】. In benchmarks, on a 48 GB M3 Max (800 GB/s memory, 17.5 GB/s SSD), Flash-MoE loaded the 209 GB model and achieved **4.36 tokens/sec** (4-bit precision, optimized dequant)【131†L1-L4】. With 2-bit quantization (120 GB on disk) and warm cache, it peaks ~7.05 tokens/sec【131†L1-L4】. Extrapolating to a 64 GB M2 Ultra: the same 4-bit model would fit similarly (active RAM ~6 GB【131†L1-L4】), so expect ~4–5 tokens/sec. A larger model (e.g. ~300B, ~250 GB 4-bit) might still run by virtue of page-cache streaming, at perhaps slightly reduced tokens/sec (due to more frequent I/O). An M4 Ultra (192 GB, likely faster NVMe) could conceivably push model size to ~500B (≈400 GB 4-bit) at ~5–7 tokens/sec. Beyond these, SSD throughput (~16–20 GB/s on high-end Mac SSD) and CPU/GPU dequantization become bottlenecks【124†L96-L104】【131†L1-L4】. In summary: on a 64 GB Apple device, FlashMoE unlocks *hundreds-of-billions* parameter models at a few tokens/sec. (Confidence: 75%–85% – based on actual 397B results【131†L1-L4】 and hardware specs【129†L79-L84】.)  

## Cache Policy & Ceiling  
The cache replacement in FlashMoE is essentially **an ML-informed hybrid** of recency and frequency【133†L65-L72】. In the Apple implementation, however, the native page cache handles eviction (old pages are dropped when memory is needed). We observed *no custom eviction engine* on macOS – simply letting the OS evict the least-recently-used pages. This still achieves high reuse for MoE’s sparsity: when an expert is activated at a token, its parameters are likely still resident until that layer’s next use. FlashMoE’s authors report *51% higher cache-hit* than LRU, meaning the learned policy would normally preload or retain frequently-used experts【133†L65-L72】. On Apple’s UMA, we rely on the system’s own LRU; in practice the hit rate was ~71%【124†L96-L104】, so any minor loss compared to an ML policy is offset by simplicity.  

**Performance Ceiling:** Ultimately, throughput is limited by I/O and compute. Each token requires loading ~4 expert blocks (~27 MB 4-bit). Even with perfect caching, a maximum SSD bandwidth of ~17 GB/s translates to ~630 tokens/s of data, but real-world FMA/GPU compute and serialization reduce this to single digits of tokens/sec. The demonstrated 4.4 tps (4-bit) corresponds to ~30 MB/s of effective load, well below peak I/O, indicating the pipeline (GPU computation + I/O) is balanced.  We estimate that around **400–500B parameters** (4-bit) is the upper limit for a 64–192 GB machine: beyond that, simply fetching new experts would saturate the SSD and slow throughput unacceptably. (Confidence ~75%.) In terms of active parameters, MoE sparsity means only ~17–20B are hot at once (as shown by the 397B model only needing ~6 GB RAM【131†L1-L4】). Thus, the *unified memory footprint* remains moderate, and the ceiling is set by SSD+CPU. 

**Conclusion:** FlashMoE’s SSD-based expert swapping enables *hundreds-of-billions*-parameter MoE inference on single Apple machines【124†L96-L104】【131†L1-L4】. The core trick is offloading experts to NVMe and letting macOS manage caching (no custom LRU needed)【124†L96-L104】. On a 64 GB M2/M4 Ultra, we expect similar ~4–5 tokens/sec for large MoEs (4-bit) and slightly higher (~7–10 tps) for narrower models or 2-bit quantization. Going beyond ∼400B parameters would face diminishing returns as SSD I/O becomes the bottleneck. (Confidence ≥ 75% for all estimates.)
