TurboQuant — What It Actually Is
2-stage KV cache compression. Stage 1 (PolarQuant): random orthogonal rotation normalizes KV vectors for mathematically optimal quantization. Stage 2 (QJL): 1 residual bit corrects compression error.

Results: 4-6x memory reduction. At 4-bit: indistinguishable from FP16 on 3B+ parameter models. 8x throughput on H100 GPUs. Works on any transformer, no retraining required.

Open-source implementations already on GitHub (PyTorch, Triton kernels, vLLM integration). ICLR 2026 paper.
FRAMING NOTE: TurboQuant operates at the inference hardware layer. When using Claude via API, Anthropic's servers manage KV caches — not Velorin. TurboQuant becomes directly relevant when running LOCAL models (Apple M-series 36GB build). For the current API-based architecture: this layer is handled for you. Park TurboQuant until the local inference build begins.

The Technologies — What Each One Actually Does
RAG (flat chunk retrieval) Graph RAG (relationship retrieval) Vector Database (semantic search index) Pointer Traversal (current Brain approach) TurboQuant compression (at retrieval time for local)
The Real Difference — Not Competing, Complementary
RAG (Retrieval Augmented Generation): Embeds documents as vectors, stores in a vector DB, retrieves by semantic similarity. Finds chunks that "sound like" your query. Good for: "what does Jiang's research say about X?" Bad for: understanding relationships and structure. Retrieves fragments, not architecture.

Graph RAG (Microsoft, 2024): Converts documents into a knowledge graph first — extracts entities and relationships using LLM calls. Then retrieves subgraphs, not just chunks. Good for: "how does X relate to Y through Z?" Expensive to build (many LLM calls to extract). More powerful than flat RAG for relationship-heavy queries. Open-sourced by Microsoft.

Vector Database: The infrastructure layer that makes RAG possible. Not a strategy — a tool. pgvector (Postgres), Chroma, Qdrant, Pinecone, Weaviate. Mature, commodity technology. You choose one based on scale and hosting preference.

Pointer Traversal (current Velorin Brain): Start at a named neuron, follow rated pointers to related neurons. This IS a knowledge graph — it's Graph RAG done manually. Works perfectly when you know which neuron to start from. Gap: no way to FIND the entry neuron by semantic similarity when you don't know its name.

Obsidian: NOT a retrieval technology. It is a human-facing visualization layer for linked markdown files. The Velorin Brain's file structure IS Obsidian-compatible already. Obsidian would add: graph visualization UI, backlink tracking, search. Relevant for Christian to visually navigate the Brain — not for agent-facing retrieval.
The Decision Tree
The Velorin Brain already IS a knowledge graph (like Graph RAG). It is manually maintained and pointer-traversed. The only thing it lacks is a semantic search entry point — a way to find the right starting neuron when you don't know its name.

Option A (Recommended path pending research): Add a lightweight vector index as a search entry point only. The index stores neuron names + one-line summaries as embeddings. Query = semantic search → returns neuron name → agent traverses Brain from there. Vector index does NOT replace the Brain. It is a search interface.

Option B: Add Graph RAG automatic extraction for new research documents. Automatically pull entities and relationships from Jiang's research papers into the graph. More powerful but more expensive and complex to maintain.

Option C: Keep pointer traversal only. Accept that well-named neurons and good _index.md files are sufficient. Simpler. Breaks at scale (~50-80 topics).
THIS IS THE BLOCKING DECISION. Everything above Layer 2 is built or has a clear path. Layer 2 is the one that needs a research pass and then a decision. The bottom-of-funnel research mandate targets this layer specifically.

KNOWLEDGE GRAPH / BRAIN 4 GAPS
The organized structure of knowledge with relationships. The Velorin Brain.
BUILT

Technologies
Velorin Brain (neurons + rated pointers) Region → Area → Neurons hierarchy _index.md navigation files Reverse pointer index (CA3 equivalent) Salience scoring gate Pointer renormalization Scratch log purge Obsidian (visualization — not yet connected)
Velorin Status
The Brain is structurally sound and operationally used. Its architecture is independently convergent with the hippocampal indexing model (Teyler & DiScenna) — hippocampus stores a pointer to distributed cortical pattern, not the pattern itself. The Velorin Brain IS this architecture in software.
The 4 Gaps (from HumanBrainLayers research, 92% confidence)
  • Reverse pointer index (HIGHEST PRIORITY) — biological CA3 performs pattern completion from partial cues. The Brain has no equivalent. You can follow pointers forward but cannot find "what points TO this neuron." Adding this transforms recall from one-directional traversal to genuine associative search.
  • Salience scoring gate — the brain gates what gets stored based on importance signals. The Brain stores everything equally. A salience score would prevent low-value neuron proliferation.
  • Pointer renormalization — pointer ratings (1-10) drift over time as the graph grows. No mechanism to renormalize them relative to network changes.
  • Scratch log purge — working memory equivalent. Temporary neurons from active sessions should be purged or consolidated. No automated mechanism.
None of these require rearchitecting. All four are additions, not rebuilds. The reverse pointer index is the highest priority — it compounds the value of everything else in the Brain.

The FollowBench Ceiling — Known Issue
InstructionLanguageArchitecture research (Session 015, 82% confidence) confirmed: frontier models comply reliably with approximately 3 simultaneous behavioral constraints. Rules 4+ in any boot file are operating probabilistically, not reliably.

Every Velorin agent boot file exceeds 3 constraints. This means the current boot architecture is partially decorative above the ceiling. We do not know which rules are below the ceiling (reliable) and which are above it (probabilistic).

Fix requires: structural enforcement via AgentSpec or bootloader pattern — not adding more behavioral rules. This is an architecture session, not a rules editing session.
